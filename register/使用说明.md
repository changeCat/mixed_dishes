# Docker 使用说明

## 1. 文件说明

本目录下的 Docker 运行文件如下：

- `Dockerfile`：用于构建运行 `chatgpt_register.py` 的 Docker 镜像
- `config.json`：脚本默认配置文件
- `chatgpt_register.py`：主执行脚本
- `chatgpt_register.py.bak`：脚本备份文件

## 2. 先配置 [`register/config.json`](register/config.json)

脚本已改为**非交互运行**，启动后不会再询问代理、CPA 清理、注册数量、并发数等内容，统一从 [`register/config.json`](register/config.json) 或环境变量读取。

### 2.1 参数说明表

| 参数名 | 默认值 | 说明 |
| --- | --- | --- |
| `_comment` | `ChatGPT 批量自动注册工具配置文件 (DuckMail 临时邮箱版)` | 配置文件注释说明，无业务影响 |
| `total_accounts` | `3` | 每次运行要注册的账号数量 |
| `max_workers` | `3` | 并发线程数，实际执行时不会超过 `total_accounts` |
| `duckmail_api_base` | `https://api.duckmail.sbs` | DuckMail API 基础地址 |
| `duckmail_bearer` | `""` | DuckMail 授权令牌，必填；为空时通常无法正常注册 |
| `proxy` | `127.0.0.1:7890` | 代理地址；有值则使用，留空则不使用代理 |
| `output_file` | `registered_accounts.txt` | 注册结果输出文件名 |
| `enable_oauth` | `true` | 是否执行 OAuth 并获取 Codex Token |
| `oauth_required` | `true` | 当 OAuth 失败时是否将整个注册流程判定为失败 |
| `oauth_issuer` | `https://auth.openai.com` | OAuth 认证服务地址 |
| `oauth_client_id` | `app_EMoamEEZ73f0CkXaXp7hrann` | OAuth 客户端 ID |
| `oauth_redirect_uri` | `http://localhost:1455/auth/callback` | OAuth 回调地址 |
| `ak_file` | `ak.txt` | Access Key 输出文件名 |
| `rk_file` | `rk.txt` | Refresh Key 输出文件名 |
| `token_json_dir` | `codex_tokens` | Token JSON 输出目录 |
| `upload_api_url` | `""` | CPA 上传接口地址；为空则跳过上传与清理相关流程 |
| `upload_api_token` | `""` | CPA 上传接口认证 Token |
| `cpa_cleanup_enabled` | `true` | 是否在注册前执行 CPA 清理；只有配置了 `upload_api_url` 才会实际生效 |
| `continue_on_missing_duckmail_bearer` | `false` | 未配置 `duckmail_bearer` 时是否继续执行；建议保持 `false` |
| `telegram_bot_token` | `""` | Telegram Bot Token；用于发送运行结果通知 |
| `telegram_chat_id` | `""` | Telegram 接收消息的聊天 ID |
| `telegram_notify_enabled` | `false` | 是否启用 Telegram 推送；启用后会在流程结束时发送汇总消息 |

### 2.2 常用配置建议

- 只要 `proxy` 有值，脚本就会使用代理；如果不想走代理，请把 `proxy` 改成空字符串 `""`
- 如果不需要 CPA 上传与清理，可保持 `upload_api_url` 和 `upload_api_token` 为空
- 如果只是注册账号、不关心 OAuth Token，可将 `enable_oauth` 设为 `false`
- 如果要启用 Telegram 推送，请同时配置 `telegram_notify_enabled=true`、`telegram_bot_token` 和 `telegram_chat_id`
- Docker 定时场景建议提前在 [`register/config.json`](register/config.json) 中写好参数，避免运行时再传过多环境变量

## 3. 构建镜像

先进入项目根目录 `mixed_dishes`，然后执行：

```bash
docker build -t chatgpt-register:latest ./register
```

说明：

- `chatgpt-register:latest` 是镜像名称和标签
- `./register` 是 Docker 构建上下文目录

## 4. 手动运行一次

不带环境变量直接运行：

```bash
docker run --rm --name chatgpt-register-job chatgpt-register:latest
```

说明：

- `--rm`：容器执行结束后自动删除
- `--name chatgpt-register-job`：指定容器名称

## 5. 使用环境变量覆盖配置

脚本会优先读取环境变量，再读取 `config.json` 中的默认值。

例如：

```bash
docker run --rm \
  --name chatgpt-register-job \
  -e DUCKMAIL_BEARER=your_token \
  -e PROXY=http://host.docker.internal:7890 \
  -e TOTAL_ACCOUNTS=3 \
  -e MAX_WORKERS=3 \
  -e CPA_CLEANUP_ENABLED=true \
  chatgpt-register:latest
```

常用环境变量：

- `DUCKMAIL_API_BASE`
- `DUCKMAIL_BEARER`
- `PROXY`
- `TOTAL_ACCOUNTS`
- `MAX_WORKERS`
- `ENABLE_OAUTH`
- `OAUTH_REQUIRED`
- `OAUTH_ISSUER`
- `OAUTH_CLIENT_ID`
- `OAUTH_REDIRECT_URI`
- `AK_FILE`
- `RK_FILE`
- `TOKEN_JSON_DIR`
- `UPLOAD_API_URL`
- `UPLOAD_API_TOKEN`
- `CPA_CLEANUP_ENABLED`
- `CONTINUE_ON_MISSING_DUCKMAIL_BEARER`
- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`
- `TELEGRAM_NOTIFY_ENABLED`

## 6. Linux 使用 cron 每天定时执行

例如每天凌晨 2 点执行一次：

```cron
0 2 * * * docker run --rm --name chatgpt-register-job -e DUCKMAIL_BEARER=your_token -e PROXY=http://host.docker.internal:7890 -e TOTAL_ACCOUNTS=3 -e MAX_WORKERS=3 -e CPA_CLEANUP_ENABLED=true chatgpt-register:latest >> /var/log/chatgpt-register.log 2>&1
```

添加方法：

```bash
crontab -e
```

把上面的定时任务追加进去并保存即可。

## 7. Windows 任务计划程序定时执行

如果你在 Windows 上运行 Docker，也可以使用任务计划程序。

命令行创建每日 02:00 任务示例：

```cmd
schtasks /Create /SC DAILY /ST 02:00 /TN ChatGPTRegisterDaily /TR "docker run --rm --name chatgpt-register-job -e DUCKMAIL_BEARER=your_token -e PROXY=http://host.docker.internal:7890 -e TOTAL_ACCOUNTS=3 -e MAX_WORKERS=3 -e CPA_CLEANUP_ENABLED=true chatgpt-register:latest" /F
```

## 8. 推荐执行顺序

建议按以下顺序操作：

1. 先修改 [`register/config.json`](register/config.json) 中的参数
2. 执行镜像构建命令
3. 手动运行一次确认脚本可正常执行
4. 配置 `cron` 或 Windows 任务计划程序
5. 观察日志输出，确认每天任务正常执行

## 9. 当前逻辑说明

当前脚本中的运行控制逻辑已经调整为：

- 代理不再交互输入，直接读取 [`register/config.json`](register/config.json) 中的 `proxy`
- 如果 `proxy` 有配置则使用，否则不使用
- CPA 清理不再交互确认，直接读取 [`register/config.json`](register/config.json) 中的 `cpa_cleanup_enabled`
- 注册数量不再交互输入，直接读取 [`register/config.json`](register/config.json) 中的 `total_accounts`
- 并发数不再交互输入，直接读取 [`register/config.json`](register/config.json) 中的 `max_workers`
- 未配置 `duckmail_bearer` 时是否继续执行，直接读取 [`register/config.json`](register/config.json) 中的 `continue_on_missing_duckmail_bearer`
